Apply the following hardening fixes and quality improvements to the existing Stripe Billing Integration implementation for Iron Metrics.

Requirements:

1. Log webhook failures into the integration timeline
   Problem:
   Webhook failures are tracked in stripeWebhookEvents with status = "failed" but are not written to stripeIntegrationEvents, so the frontend Integration Timeline does not show webhook issues.

   Fix:
   - In the webhook processing error path, create a stripeIntegrationEvent with eventType = "webhook_failure"
   - Include safe structured details such as:
     - webhook event type
     - stripe event id
     - gym id
     - failure reason/message
     - processing timestamp
   - Do not include secrets or raw sensitive payload data
   - Ensure webhook failures appear in the Integration Timeline UI

2. Require a dedicated production encryption key
   Problem:
   STRIPE_ENCRYPTION_KEY currently falls back to DATABASE_URL.slice(0,32) or a hardcoded default string. This is not acceptable for production-grade secret management.

   Fix:
   - Remove insecure fallback behavior for production use
   - Require a dedicated STRIPE_ENCRYPTION_KEY environment variable for encryption/decryption of Stripe API keys
   - If the key is missing in environments where Stripe connection actions are attempted, fail safely with a clear server-side error
   - Add startup validation or guarded runtime validation so this is easy to diagnose
   - Keep behavior secure and explicit
   - Do not expose the encryption key to client-side code

3. Clarify and correct the misleading billingLinked metric
   Problem:
   billingLinked currently measures matched customers / total customers, but the naming implies linked billing records.

   Fix:
   - Rename the backend metric and related frontend labels to reflect what is truly measured
   - Distinguish clearly between:
     - customer match coverage
     - billing record match coverage
   - If both metrics are useful, return both
   - Update UI copy so the owner/admin sees accurate terminology
   - Avoid misleading labels like “Member Matching” if the denominator is not billing records
   - Keep the language professional and precise

4. Add a dry-run safety gate before first live sync
   Problem:
   Users can run a live sync immediately without first running a dry run.

   Fix:
   - Require at least one successful dry run before the first live sync
   - If no dry run exists yet, disable or block live sync and show a clear explanation
   - Add a confirmation dialog before the first live sync even after dry-run succeeds
   - Allow subsequent live syncs without unnecessary friction, but still keep an optional confirmation where appropriate
   - Reflect this state in the onboarding checklist/readiness UI

5. Optimize verifyMatchBelongsToGym lookup
   Problem:
   verifyMatchBelongsToGym currently loads all matches for the gym and searches in memory, which is inefficient at scale.

   Fix:
   - Replace this with a direct storage/database query using match id + gym id
   - Add a dedicated storage method such as getStripeCustomerMatchByIdAndGym(id, gymId)
   - Ensure all existing tenant validation behavior remains intact
   - Preserve current authorization and safety semantics while improving performance

   Problem:

   Fix:
